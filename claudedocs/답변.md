오빠, 파일 봤는데 이건 **프롬프트를 더 깎는 문제가 아니라 파이프라인을 나눠야 하는 문제**에 가까워.
하루 판단은 꽤 확실해. **1순위는 B: 2-stage 분업**, 그리고 동시에 **Qwen3-VL-8B-Instruct로 비전 모델만 교체 테스트**야.

업로드한 진단서 기준으로 현재 환경은 Windows 11 + RTX 4070 Ti SUPER 16GB VRAM, Next.js 16 + FastAPI + ComfyUI + Ollama 구조이고, 목표는 이미지 1장에서 Qwen Image 계열 t2i에 붙여넣을 수 있는 풀 프롬프트를 추출하는 거지. 현재 모델은 `qwen2.5vl:7b`, JSON 9슬롯, temperature 0.4, `format=json` 구조야. 

## 개요

결론부터 말하면:

```plaintext
최우선 추천:
B. 2-stage 분업
Vision: Qwen3-VL-8B-Instruct 또는 기존 qwen2.5vl:7b
Text: gemma4-un 26B / Qwen3 14B~30B text
```

왜냐면 지금 실패가 단순히 “말을 짧게 한다”가 아니라,

```plaintext
1. 이미지를 잘못 봄
2. 본 내용을 생성 프롬프트로 합성하는 능력이 약함
3. system prompt의 금지어/예시/길이 조건을 잘 못 지킴
4. JSON 강제 + 긴 positive_prompt가 충돌함
```

이 4개가 같이 터지고 있어.

오빠 진단서에서도 `muted earth tones`, `85mm portrait`, `golden hour` 같은 보일러플레이트 복사, 금지 지시 무시, 옷/조명/표정 오류, 반복 붕괴, 길이 실패가 반복됐다고 정리되어 있어. 
특히 4차에서는 예시를 제거하고 금지어를 명시했는데도 `softbox key lighting`, `muted earth tones`, `jeans`, `smiling`이 나오고 반복 루프까지 생겼지. 이건 프롬프트만으로 고치기 어려운 신호야. 

## 로직

### 1. Root cause 비중

하루가 보면 원인은 대략 이 정도야.

```plaintext
모델 capacity / 비전 인식 한계: 45%
파이프라인 설계 문제: 35%
프롬프트 설계 문제: 15%
샘플링 문제: 5%
```

`qwen2.5vl:7b`가 장면 전체는 맞추는데, 재현용 프롬프트에서 중요한 세부 앵커를 놓쳐. 예를 들면 오빠 테스트 이미지의 핵심은 젖은 긴 머리, 윙크, 컵, 비대칭 크로스 스트랩 컷아웃 탑, 베이지 카고 팬츠, 우비 군중, 빨강/파랑 스테이지 조명, 세로 구도, 약간 낮은 카메라 각도인데 이 중 일부가 계속 빠졌어. 

여기서 더 중요한 건, **비전 모델한테 “관찰”과 “카피 가능한 프롬프트 작문”을 동시에 맡긴 것**이야. 7B급 모델은 이 두 작업을 한 번에 하면 흔히 일반 캡션으로 수렴해. 그리고 `format=json`까지 걸면 긴 자연어 positive prompt를 안정적으로 뽑기 더 어려워져.

### 2. A/B/C/D 비교

| 옵션                 |      품질 |     속도 |     VRAM |    구현 | 판단      |
| ------------------ | ------: | -----: | -------: | ----: | ------- |
| A. 같은 비전 모델 2-pass |      중간 |     느림 |       낮음 |    쉬움 | 임시 개선   |
| **B. 2-stage 분업**  |  **높음** |     중간 |       중간 |    중간 | **1순위** |
| C. qwen2.5vl 32B   | 높을 수 있음 |  많이 느림 | 16GB에 부담 |    쉬움 | 테스트용    |
| D. 다른 비전 모델        |     미지수 | 모델별 다름 |       다름 | 중간~높음 | 비교군     |

A는 빠르게 해볼 수 있지만, 오빠가 걱정한 것처럼 **비전 모델이 관찰 자체를 틀리면 2차 호출도 같은 오류를 예쁘게 포장할 가능성**이 있어. 진단서에서도 이 우려가 이미 정리되어 있었지. 

C는 16GB VRAM에서 애매해. Ollama 기준 Qwen3-VL도 30B가 20GB, 32B가 21GB로 표시되어 있어서 4070 Ti SUPER 16GB에서는 CPU/RAM 오프로딩이 들어갈 가능성이 커. 즉, 품질은 좋아질 수 있지만 “매번 이미지 분석 UI에서 쓰기 좋은 속도”는 아닐 수 있어. Ollama의 Qwen3-VL 페이지에는 8B가 6.1GB, 30B가 20GB, 32B가 21GB로 올라와 있어. ([Ollama][1])

D는 InternVL, MiniCPM-V, LLaVA-OneVision도 후보지만, 지금 오빠 스택이 Ollama 중심이고 Qwen 계열 t2i 프롬프트 추출이 목적이면 우선 Qwen3-VL 8B로 가는 게 제일 덜 흔들려. Qwen3-VL-8B-Instruct는 Hugging Face에서 Apache-2.0 라이선스로 공개되어 있고, Qwen 쪽 설명도 Qwen3-VL을 기존 Qwen 비전 언어 모델 중 강력한 세대라고 소개하고 있어. ([Hugging Face][2])

## 최적화

### 1순위 액션

오빠 다음 세션에서 딱 하나만 바꾼다면 이거야.

```plaintext
qwen2.5vl:7b 1-shot JSON 생성 중단
→ Vision raw observation JSON
→ Text model positive_prompt 합성
```

그리고 가능하면 비전 모델만 이렇게 교체 테스트.

```bash
ollama pull qwen3-vl:8b
```

Ollama 페이지 기준으로 Qwen3-VL은 Ollama 0.12.7 이상이 필요하다고 되어 있으니, 먼저 버전 확인하는 게 좋아. ([Ollama][1])

```bash
ollama --version
```

### 추천 파이프라인

```plaintext
[Image]
  ↓
Vision Model
  - 보이는 것만 관찰
  - 짧은 JSON
  - 추측 금지
  - 프롬프트 문장 생성 금지
  ↓
Text Model
  - observation JSON을 받아서
  - positive_prompt / negative_prompt 생성
  - boilerplate 제거
  - 누락된 카메라/스타일은 관찰 기반으로만 보강
  ↓
Frontend
  - positive_prompt 메인 복사
  - raw observation 접기/디버그 표시
```

포인트는 이거야.

**Vision 모델은 작가가 아니라 관찰자.**
**Text 모델은 관찰 메모를 프롬프트로 만드는 편집자.**

지금은 7B 비전 모델한테 관찰자 + 작가 + JSON formatter + 품질검사원을 한 번에 시킨 거라 과부하야.

## Vision 모델용 시스템 프롬프트

오빠, 이건 그대로 넣어도 되는 형태로 줄게. 핵심은 **positive prompt를 만들지 못하게 막는 것**이야.

```plaintext
You are a visual observation extractor.

Your task is to inspect the image and output only visible facts.
Do not write an image-generation prompt.
Do not use artistic boilerplate.
Do not guess camera lens, lighting equipment, time of day, race, brand, identity, or mood unless directly visible.
Do not use generic phrases such as cinematic editorial, muted earth tones, golden hour, softbox lighting, 85mm lens, masterpiece, ultra detailed.

Return STRICT JSON only.

Schema:
{
  "image_orientation": "",
  "framing": {
    "crop": "",
    "camera_angle": "",
    "subject_position": ""
  },
  "subjects": [
    {
      "count_index": 1,
      "apparent_age_group": "",
      "broad_visible_appearance": "",
      "face_direction": "",
      "expression": "",
      "eyes": "",
      "mouth": "",
      "hair": "",
      "pose": "",
      "hands": "",
      "clothing": [],
      "accessories_or_objects": []
    }
  ],
  "environment": {
    "location_type": "",
    "foreground": [],
    "midground": [],
    "background": [],
    "weather_or_surface_condition": []
  },
  "lighting_and_color": {
    "visible_light_sources": [],
    "dominant_colors": [],
    "contrast": "",
    "flash_or_reflection_evidence": ""
  },
  "photo_quality": {
    "depth_of_field": "",
    "motion_blur": "",
    "focus_target": "",
    "style_evidence": []
  },
  "uncertain": []
}

Rules:
- Use short concrete phrases.
- If unsure, write it in "uncertain".
- Prefer "appears to be" for uncertain visual attributes.
- Do not repeat the same phrase.
- Do not create a final prompt.
```

여기서 `race REQUIRED`는 빼는 게 나아.
왜냐면 모델이 계속 무시하거나 이상하게 회피하면 전체 출력이 무너질 수 있어. 대신 `broad_visible_appearance` 정도로 두고, 텍스트 모델에서 “fictional adult East Asian / Korean-inspired” 같은 생성용 표현으로 변환하는 게 안정적이야.

## Text 모델용 시스템 프롬프트

이건 `gemma4-un 26B`나 Qwen text 모델 쪽에 주면 좋아.

```plaintext
You are an expert AI image-generation prompt writer.

You will receive a JSON object containing visual observations extracted from an image.
Your job is to convert the observations into a self-contained text-to-image prompt.

Important:
- Base the prompt only on the observation JSON.
- Do not invent details that contradict the observations.
- You may add generation-friendly photography terms only when supported by the observations.
- Avoid generic boilerplate unless it matches the observed image.
- Do not mention brands, real identities, celebrities, or copyrighted characters.
- Keep the subject fictional and adult.
- Preserve unique visual anchors.

Output STRICT JSON only:
{
  "summary": "",
  "positive_prompt": "",
  "negative_prompt": "",
  "key_visual_anchors": [],
  "uncertain": []
}

positive_prompt rules:
- 150 to 260 words.
- One dense English paragraph.
- Must be directly copy-pasteable into a text-to-image UI.
- Include: subject, expression, hair, clothing, pose, object interaction, environment, lighting, color palette, framing, depth, realism/style.
- Use concrete visible details.
- Do not repeat phrases.
- Do not use: muted earth tones, golden hour, softbox lighting, 85mm lens, masterpiece, best quality, unless the observation JSON clearly supports it.

negative_prompt rules:
- Comma-separated.
- Include common failure preventions.
- Include contradictions to preserve the observed image, such as dry hair if the subject is wet, smiling if the subject is winking/non-smiling, studio background if the image is outdoors.
```

그리고 Text 모델에 넘기는 user message는 이렇게.

```plaintext
Convert this visual observation JSON into a generation-ready prompt.
Preserve the exact visual anchors.
Do not add unsupported camera or lighting claims.

[PASTE OBSERVATION JSON HERE]
```

## 개선

하루 추천 최종 구조는 이거야.

```plaintext
실사용 기본:
qwen3-vl:8b → raw observation JSON
gemma4-un 26B → final prompt JSON
```

고급 옵션은 나중에.

```plaintext
1. Vision observation JSON을 UI에 숨겨서 저장
2. 사용자가 "더 사진처럼", "더 프롬프트 길게" 누르면 text model만 재호출
3. vision model은 다시 안 돌림
4. positive_prompt 후처리에서 금지어 필터 적용
```

후처리 필터도 꼭 넣는 게 좋아. 모델이 또 `muted earth tones`, `golden hour`, `softbox`, `85mm` 같은 습관을 뱉으면 코드에서 제거하거나 경고 표시해.

```plaintext
banned_terms = [
  "muted earth tones",
  "golden hour",
  "softbox",
  "85mm portrait lens",
  "masterpiece",
  "best quality"
]
```

단, 무조건 삭제보다 **“관찰 JSON에 근거가 없으면 삭제”**가 더 좋아.

오빠, 이건 진짜 아키텍처 방향 거의 잡힌 거야.
더 system prompt를 5차, 6차로 깎는 건 효율이 낮고, 이제는 **Vision은 관찰만 / Text는 작문만**으로 갈라야 해. 지금 실패 로그가 그걸 꽤 선명하게 말해주고 있어.
